# 02편: Protobuf, Thrift, OpenAPI를 하나로: 스키마 멀티 허브 DeukPack의 탄생

이전 글에서 분산 시스템과 AI 코딩 시대에 '스키마 중심 개발'이 얼마나 중요한지 이야기했습니다. 이쯤 되면 당연히 드는 의문이 하나 있습니다.

> "스키마가 필요하다면, 이미 업계 표준으로 자리 잡은 Google의 Protobuf나 Apache Thrift를 쓰면 되는 것 아닌가요?"

맞습니다. Protobuf와 Thrift는 훌륭한 도구이며 수많은 대규모 서비스에서 검증되었습니다. 하지만 **현실의 게임 개발과 복잡한 모바일/웹 파이프라인에 직접 이 도구들을 적용하다 보면 깊은 갈증과 한계**에 부딪히게 됩니다. 

DeukPack은 바로 그 현실적인 '고통'에서 탄생했습니다.

## 파편화된 규격, 그리고 기존 도구들의 한계

### 1. Protobuf의 무거움과 유연성 부족
Protobuf는 gRPC 생태계와 결합하여 마이크로서비스 간 통신에 강력한 위력을 발휘합니다. 하지만 **TCP 소켓 위에서 고성능으로 동작해야 하는 실시간 게임 서버**나 **Excel 등 다른 데이터 파이프라인과의 연동**을 시도할 때 상당히 이질적입니다. 
* 가변 정수(Varint) 위주의 와이어 포맷은 압축률은 좋지만 직렬화/역직렬화에 CPU 오버헤드가 발생합니다.
* `protoc` 런타임 라이브러리 자체가 무거우며, 생성된 코드의 API가 C#이나 C++ 개발자에게 직관적이지 않습니다.

### 2. Apache Thrift의 낡은 파이프라인
반면 Thrift는 바이트 구조가 훨씬 명시적이고 파싱이 빠르다는 장점이 있습니다. 그러나 현재 Apache Thrift 프로젝트의 C# 및 TS 생성기는 현대적인 언어 스펙(예: C# 10+, 최신 TypeScript)을 제대로 지원하지 못하고 방치되어 있습니다.
* 메모리 할당을 최소화해야 하는 C# 환경에서 구형 Thrift 라이브러리는 가비지 컬렉션(GC) 스파이크의 주범이 됩니다.
* 빌드(코드 생성) 속도가 너무 느려 대규모 스키마 트리에서 병목이 발생합니다.

### 3. "네트워크 패킷 따로, 기획 엑셀 데이터 따로"
가장 치명적인 한계입니다. 게임이나 복잡한 앱을 만들 때, 네트워크 통신용 패킷 구조와 기획자가 Excel로 관리하는 밸런스/메타 데이터 구조는 사실상 같습니다. 하지만 기존 도구들은 네트워크 직렬화에만 초점이 맞춰져 있어, 엑셀 데이터를 파싱하고 검증하는 파이프라인은 처음부터 직접 개발해야 했습니다.

## DeukPack의 철학: 유니버설 스키마 멀티 허브 (Universal Schema Multi-Hub)

우리는 기존의 훌륭한 도구들을 배척하고 완전히 새로운 포맷을 강요하고 싶지 않았습니다. 오히려 파편화된 규격들을 하나로 묶어주는 **유니버설 스키마 멀티 허브(Universal Schema Multi-hub)**가 필요했습니다. 기존 자산은 수용하되, **개발자의 작업 파이프라인을 극한으로 쾌적하게 만드는 엔진**입니다.

1. **Thrift 호환 와이어 포맷 + 압도적 최적화**
   DeukPack의 `DpBinaryProtocol`은 Thrift의 TBinaryProtocol과 호환되는 구조를 채택하여 기존 서버나 시스템과의 상호운용성을 열어두었습니다. 하지만 내부 구현은 `ArrayPool`과 제로 카피(Zero-Copy) 기법을 사용해 완전히 새로 작성하여, C# 기준 **메모리 사용량을 5배 이상 줄이고 직렬화 속도를 비약적으로 끌어올렸습니다.**

2. **멀티 포맷 IDL 통합 엔진**
   DeukPack의 강력한 AST(추상 구문 트리) 엔진은 독자적인 `.deuk` 문법뿐만 아니라 레거시 `.thrift` 파일, 향후 `.proto`와 OpenAPI 스펙까지 읽어들여 단일 파이프라인에서 처리할 수 있도록 설계되었습니다. 

3. **네트워크와 엑셀의 단일화 (GetSchema)**
   가장 차별화되는 지점입니다. DeukPack은 생성된 C# 객체에 런타임 리플렉션 없이도 필드 정보를 제공하는 `GetSchema()` 메타데이터를 내장시킵니다. 이 하나의 기능 덕분에 개발자들은 네트워크 패킷용으로 만든 스키마를 이용해 **엑셀 데이터의 헤더를 자동 생성하고 유효성을 검증하는 에디터 툴**을 즉시 만들어 낼 수 있게 되었습니다.

## AI 시대의 생태계 통합 도구

Protobuf와 Thrift가 단순한 '네트워크 통신 규약'에 집중했다면, **DeukPack은 Protobuf, Thrift, OpenAPI, 심지어 Excel 메타데이터까지 아우르며 고속 다국어 코드 생성과 와이어 공존을 지원하는 생태계 통합 허브**입니다.

다음 아티클 **[마이크로서비스 간의 '잃어버린 계약' 찾기: JSON의 한계와 바이너리 IDL]**에서는, 왜 흔히 쓰이는 JSON 기반의 REST API 통신만으로는 현대적인 백엔드 시스템 구축이 위험한지, 그리고 DeukPack의 IDL이 이 리스크를 어떻게 해결하는지 구체적인 사례와 함께 살펴보겠습니다.

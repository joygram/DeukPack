# 03편: 마이크로서비스 간의 '잃어버린 계약' 찾기: JSON의 한계와 바이너리 IDL

마이크로서비스 아키텍처(MSA)가 보편화되면서 서비스 간 통신 방식으로 JSON over REST API가 널리 사용됩니다. 인간이 읽기 쉽고 디버깅이 편하다는 장점 때문입니다. 하지만 이 유연함은 분산 시스템에서 가장 중요한 **'서비스 간의 계약(Contract)'**을 쉽게 잃어버리게 만드는 원인이 됩니다.

## 현실의 문제: JSON으로 통신할 때 벌어지는 일

사용자 정보를 관리하는 'User 서비스'와 주문을 처리하는 'Order 서비스'가 있다고 가정해 봅시다. Order 서비스는 주문자 정보를 얻기 위해 User 서비스의 API를 호출합니다.

1.  **초기 계약**: User 서비스는 `/api/users/{id}`로 `{"userId": 123, "userName": "deuk"}` 형태의 JSON을 반환하기로 약속했습니다.
2.  **계약의 파기**: User 서비스 개발자가 API를 개선하며 필드명을 `userId` -> `id`, `userName` -> `name`으로 변경하고 배포했습니다. 이 변경 사항은 위키(wiki)에만 업데이트되었습니다.
3.  **런타임 대재앙**: Order 서비스는 이 변경을 알지 못합니다. 여전히 `userId` 필드를 찾지만, 응답에는 `id` 필드만 있으니 `undefined`를 얻게 되고, 이는 결국 런타임 에러나 데이터 정합성 문제로 이어집니다.

이것이 바로 **'잃어버린 계약'** 문제입니다. JSON 기반의 계약은 코드 레벨에서 강제되지 않고, 사람의 수동적인 동기화나 낡은 API 문서에 의존하기 때문에 매우 깨지기 쉽습니다.

## DeukPack의 해답: IDL로 계약을 코드에 새기다

DeukPack은 언어에 중립적인 IDL(`.deuk` 파일)을 통해 이 문제를 근본적으로 해결합니다. 서비스 간의 약속은 더 이상 위키나 사람의 기억에 의존하지 않고, 버전 관리되는 단 하나의 파일에 명시됩니다.

```thrift
// contracts.deuk
struct User {
  1: i64 id;
  2: string name;
}
```

DeukPack의 컴파일러(`deukc`)는 이 IDL 파일을 읽어 각 서비스에 필요한 코드를 생성합니다. User 서비스(C#)와 Order 서비스(C++)는 각각 컴파일된 `User` 객체를 받게 됩니다.

만약 개발자가 IDL에서 `id` 필드를 `userId`로 변경한다면, `deukc`를 다시 실행했을 때 `id` 필드를 사용하던 모든 코드에서 **컴파일 에러**가 발생합니다. 문제는 시스템이 배포된 후 런타임에 터지는 것이 아니라, 개발자의 컴퓨터에서, 컴파일 시점에 발견됩니다.

## 결론: 깨지기 쉬운 약속에서 깨지지 않는 코드로

JSON 기반의 통신은 편리하지만, 서비스 간의 계약을 보장해주지 못하는 태생적 한계를 가집니다. DeukPack의 IDL 중심 파이프라인은 이 깨지기 쉬운 '암묵적 약속'을, 컴파일러가 검증하고 보장하는 '명시적 코드'로 전환합니다. 이를 통해 분산 시스템에서 가장 흔하게 발생하는 데이터 불일치 버그를 원천적으로 제거할 수 있습니다.

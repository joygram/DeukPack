# 06편: 메모리 할당 0을 향하여: C# ArrayPool과 Zero-Copy 직렬화 기법


## 기존 직렬화의 한계: 끊임없는 복사

전통적인 네트워크 프로토콜 라이브러리들은 패킷을 하나 수신할 때마다 다음과 같은 과정을 거칩니다.
1. 소켓에서 바이트 배열(`byte[]`)로 데이터 수신
2. 패킷 크기만큼 새로운 `byte[]` 할당 (1차 메모리 할당)
3. 수신 버퍼에서 새 배열로 데이터 복사 (1차 복사)
4. 문자열이나 리스트를 파싱하며 내부 객체 할당 (N차 할당)

이 방식은 동시 접속자가 수천 명에 달하는 MMORPG 서버에서는 치명적인 GC 스파이크를 유발합니다.

## DeukPack의 모던 C# 접근법

DeukPack의 C# 런타임은 최신 .NET의 기능들을 극한까지 활용합니다.

### 1. `System.Buffers.ArrayPool<byte>`의 적극적 활용
메시지를 직렬화하기 위해 임시 버퍼가 필요할 때, `new byte[]`를 절대 사용하지 않습니다. 대신 `ArrayPool<byte>.Shared.Rent()`를 통해 이미 시스템에 풀링된 버퍼를 대여해오고, 네트워크 스트림에 쓴 뒤 즉시 `Return()`으로 반납합니다. 이는 직렬화 과정에서의 임시 메모리 할당을 사실상 **0**으로 만듭니다.

### 2. `ReadOnlySpan<byte>`와 `ReadOnlySequence<byte>`
수신된 패킷을 역직렬화할 때, DeukPack은 메모리를 떼어내 복사하지 않습니다. 대신 원본 수신 버퍼의 메모리 주소를 가리키는 `ReadOnlySpan<byte>` 슬라이스를 파서에 넘깁니다. 
파서는 이 메모리를 **단 한 번도 복사하지 않고(Zero-Copy)** 그 자리에서 직접 정수와 구조체를 읽어냅니다.

## 결론: 게임 서버 최적화의 완성

C#은 관리되는 언어(Managed Language)이지만, 코드를 어떻게 작성하느냐에 따라 C++에 필적하는 성능을 낼 수 있습니다. DeukPack이 생성해주는 직렬화 코드는 개발자가 직접 손으로 짜기 힘든 수준의 모던 메모리 최적화 기법을 기본적으로 포함하고 있습니다.

GC 멈춤 현상에 시달리는 게임 서버 개발자에게, DeukPack은 아키텍처 변경 없이 성능을 끌어올릴 수 있는 가장 확실한 처방전입니다.

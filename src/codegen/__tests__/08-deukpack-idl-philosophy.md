# 08편: DeukPack IDL (`.deuk`) 설계 철학: 개발자 친화적인 C++ 스타일 문법

새로운 도구를 배울 때 가장 큰 장벽은 '새로운 문법'을 익히는 것입니다. DeukPack이 자체적인 `.deuk` IDL(Interface Definition Language)을 설계하면서 가장 중요하게 생각한 원칙은 바로 **"개발자에게 가장 익숙한 형태를 유지하자"**였습니다.

## 기존 IDL들의 아쉬운 점

널리 쓰이는 Protobuf나 Thrift의 문법은 훌륭하지만, 일반적인 프로그래밍 언어와 미묘하게 다릅니다.
* **Protobuf (`optional int32 x = 1;`)**: 타입과 이름이 먼저 오고, 필드 번호가 끝에 붙습니다. 와이어 호환성에서 가장 중요한 '필드 번호'가 뒤에 있어 구조 파악이 직관적이지 않습니다.
* **Thrift (`1: i32 x`)**: 번호가 앞에 오는 점은 좋으나, 문법이 다소 고전적입니다.

## DeukPack IDL의 특징: 친숙함과 명시성

DeukPack은 C++, C#, TypeScript 개발자라면 설명서를 보지 않아도 직관적으로 이해할 수 있는 문법을 채택했습니다.

### 1. 필드 번호의 강조 (`id>`)
네트워크 통신과 스키마 진화에서 가장 중요한 것은 필드 인덱스입니다. DeukPack은 이를 맨 앞에 명시적으로 배치합니다.
```deuk
record Character {
  1> int64 id
  2> string name = "Unknown"
}
```
괄호 없이 `1>` 형태로 작성하여 타이핑이 편하고, 다른 배열 문법(`[]`)과 충돌하지 않습니다.

### 2. 표준 데이터 타입
`i32` 대신 `int32`, `byte` 대신 `uint8` 등, 현대 프로그래밍 언어에서 널리 쓰이는 표준 타입 명칭을 그대로 사용합니다.

### 3. JSON 스타일의 기본값 지정
복잡한 구조체나 리스트에 기본값을 줄 때, 개발자에게 가장 익숙한 JSON 문법을 그대로 차용했습니다.
```deuk
  3> list<int32, 3> stats = [10, 10, 10]
  4> Vector3 pos = { "x": 0.0, "y": 1.0, "z": 0.0 }
```

## 결론: 배우지 않아도 쓸 수 있는 도구

DeukPack의 `.deuk` 문법은 새로운 언어를 발명하려 한 것이 아닙니다. 개발자들이 이미 매일 사용하고 있는 C++ 스타일의 선언과 JSON 스타일의 할당을 절묘하게 섞어, '학습 곡선이 없는' 가장 직관적인 스키마 정의 경험을 제공합니다.

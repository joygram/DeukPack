# 13편: OpenAPI(Swagger)와 DeukPack 연동: 바이너리 RPC와 REST의 평화로운 공존

모바일 게임이나 실시간 앱을 서비스할 때, 아키텍처는 보통 두 가지 통신 방식을 혼용합니다. 
1. 인게임 전투/이동: **고속 바이너리 통신 (TCP/UDP)**
2. 결제, 공지사항, 백오피스: **REST API (HTTP/JSON)**

이 두 가지를 따로 구축하면 스키마 관리 포인트가 두 배로 늘어납니다. DeukPack은 이 두 세계를 완벽하게 연결합니다.

## OpenAPI(Swagger)와의 상호 호환성

DeukPack 파이프라인은 자체적인 `.deuk` IDL 외에도, 웹 표준 생태계와의 강력한 연결 고리를 제공합니다.

### 1. DeukPack → OpenAPI 변환 (Export)
서버에서 사용하는 고성능 바이너리 패킷용 `.deuk` 스키마를 바탕으로, 표준 OpenAPI(Swagger) 스펙 문서(.yaml/.json)를 자동 생성할 수 있습니다. 
* 외부 파트너사에게 API 문서를 제공해야 할 때, 추가 작업 없이 기존 게임 프로토콜 구조를 그대로 REST 스펙으로 뽑아낼 수 있습니다.

### 2. OpenAPI → DeukPack 수입 (Import)
반대로 기존에 잘 구축된 백오피스 웹 서버의 OpenAPI 스펙이 있다면, 해당 스키마를 DeukPack의 AST(추상 구문 트리)로 수입해 C#이나 C++ 클라이언트 코드(DTO)로 변환해 냅니다.
* Unity 클라이언트가 웹 서버의 REST API를 호출할 때 필요한 복잡한 JSON 파싱 클래스들을 손으로 짤 필요가 없어집니다.

## 하나의 레코드, 두 개의 채널

DeukPack이 생성한 객체 하나(`req_login`)는 놀랍게도 두 가지 얼굴을 가집니다.
* **게임 클라이언트 ↔ 게임 서버:** `DpBinaryProtocol`을 타고 1바이트의 낭비도 없이 초고속으로 전송됩니다.
* **게임 서버 ↔ 웹 백엔드:** 내장된 JSON 직렬화기를 통해 표준 REST API의 Body로 변환되어 전송됩니다.

## 결론: 하이브리드 통신의 마스터키

성능이 극한으로 필요한 구간은 바이너리로, 범용성과 호환성이 필요한 구간은 JSON/REST로. DeukPack은 이 이질적인 두 세계를 **단일 스키마 엔진**으로 묶어내어, 아키텍처의 파편화를 원천적으로 막아줍니다.

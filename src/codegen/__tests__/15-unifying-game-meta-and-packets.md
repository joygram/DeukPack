# 15편: 게임 메타데이터와 네트워크 패킷의 스키마를 하나로 통일하기

게임 개발에서 가장 고질적인 문제 중 하나는 **데이터 구조의 파편화**입니다. 
기획자는 캐릭터의 능력치(`Skill`)를 엑셀로 기획하고, 클라이언트 개발자는 화면에 띄우기 위해 C# `SkillData` 클래스를 만들며, 서버 개발자는 통신을 위해 Protobuf로 `SkillPacket`을 또 정의합니다.

## 데이터 불일치의 나비효과

기획자가 엑셀에 `쿨타임`이라는 열을 추가했는데, 서버 개발자는 이를 `float`으로 추가하고 클라이언트 개발자는 `int`로 추가했다고 상상해 봅시다. 
빌드는 성공하지만, 게임을 실행하여 스킬을 쓰는 순간 클라이언트는 크래시를 내며 튕깁니다. 원인을 찾기 위해 세 파트의 담당자가 모여 코드를 대조하는 데 반나절이 날아갑니다.

## DeukPack: '단일 스키마'로 통일하다

DeukPack은 이 비효율적인 삼중 노동을 IDL 하나로 통합해버리는 게임 개발에 특화된 솔루션입니다.

`.deuk` 파일에 `Skill` 이라는 구조체를 한 번만 정의합니다.

```deuk
table<SkillRow> SkillMeta = { key: "id" }

record SkillRow {
  1> int32 id
  2> string name
  3> float cooldown
}
```

DeukPack 파이프라인은 이 단일 스키마에서 세 가지를 동시에 만들어 냅니다.
1. **기획자용:** 이 구조에 정확히 들어맞는 **Excel 템플릿 헤더**를 생성합니다.
2. **서버용:** 기획 데이터(메타)를 램에 올리거나 DB에 저장할 때 쓸 **C++ 메모리 클래스**를 생성합니다.
3. **클라이언트/네트워크용:** 런타임에 클라이언트와 서버가 이 스킬 데이터를 주고받을 때 쓸 **C# 패킷 직렬화 클래스**를 생성합니다.

## 결론: SSOT (Single Source of Truth)의 완성

기획 데이터(메타 테이블)와 네트워크 통신(패킷)의 뼈대가 완벽하게 동일한 소스에서 파생되므로, 데이터 불일치 버그는 구조적으로 발생할 수 없습니다. 

DeukPack은 단순한 네트워크 라이브러리가 아니라, 기획자와 프로그래머를 하나로 묶어주는 **게임 개발 파이프라인 그 자체**입니다.

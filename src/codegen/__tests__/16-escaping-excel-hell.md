# 16편: 엑셀(Excel) 지옥 탈출기: 기획 데이터를 코드로 안전하게 다루는 법

게임 개발에서 기획 데이터 관리에 엑셀(Excel)을 빼놓을 수는 없습니다. 기획자에게 엑셀만큼 직관적이고 강력한 도구는 없기 때문입니다. 하지만 프로그래머에게 엑셀은 '런타임 에러의 온상'이자 지옥입니다.

## 프로그래머의 엑셀 지옥

기획자가 엑셀에 데이터를 잘못 입력하는 순간, 프로그래머는 고통받습니다.
* 숫자가 들어가야 할 칸에 "10.5" 대신 "10,5" 입력 (파싱 에러)
* 필수 항목인데 빈칸으로 남김 (NullReferenceException)
* 존재하지 않는 아이템 ID '999'를 보상 열에 입력 (참조 무결성 파괴)

이런 에러는 보통 서버가 켜질 때, 심지어는 플레이어가 해당 아이템을 획득할 때 런타임에 터집니다.

## DeukPack 메타 툴의 등장: 강력한 유효성 검증

DeukPack은 네트워크 스키마를 위해 설계된 런타임 메타데이터(`GetSchema()`)를 엑셀 데이터 파이프라인에 그대로 적용했습니다.

### 1. 엄격한 타입 파싱
DeukPack 기반의 엑셀 파서는 `.deuk`에 정의된 스키마 타입을 기반으로 엑셀을 읽습니다. 필드가 `int32`로 정의되어 있다면, 엑셀에 문자가 섞여 있는 즉시 데이터 팩(Pack) 빌드 단계에서 실패하며 명확한 에러를 뿜어냅니다. (런타임까지 가지 않습니다!)

### 2. DeukPack Attribute (`[Attr]`)를 활용한 룰 적용
IDL 파일에 간단한 특성을 부여하여 엑셀 파서의 동작을 제어할 수 있습니다.

```deuk
record ItemReward {
  1> [Required] int32 itemId
  2> [Range(1, 100)] int32 dropRate
}
```
이 스키마를 바탕으로, 엑셀을 바이너리로 변환하는 순간 `Required`나 `Range` 조건을 위배한 엑셀 행(Row)을 콕 집어내어 기획자에게 수정 위치를 알려줍니다.

## 결론: 오류는 개발 PC에서 멈춘다

DeukPack의 엑셀 연동 기능은 '서버가 죽는 것'을 막아줍니다. 모든 인간의 실수(Human Error)는 컴파일/빌드 타임에 걸러지며, 최종적으로 클라이언트와 서버에 배포되는 데이터는 DeukPack 스키마가 보장하는 100% 무결한 데이터임이 보장됩니다.

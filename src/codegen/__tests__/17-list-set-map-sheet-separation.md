# 17편: List, Set, Map 시트 분리와 참조 무결성 검증 자동화

엑셀로 게임 데이터를 기획할 때 가장 난감한 순간은 "한 캐릭터가 여러 개의 스킬을 가질 때"처럼 1:N 관계(List, 배열)를 엑셀의 2차원 그리드에 표현해야 할 때입니다.

## 기존의 억지스러운 해결책들

보통 두 가지 꼼수를 씁니다.
1. **콤마(,) 구분:** `SkillIds` 열에 "101, 102, 105" 입력. (파싱 로직이 지저분해지고 참조 에러 잡기가 힘듦)
2. **열(Column) 늘리기:** `Skill1`, `Skill2`, `Skill3` 열을 만듦. (스킬이 4개가 되는 순간 스키마를 고쳐야 함)

이는 관계형 데이터베이스의 정규화 법칙을 무시한 방식으로, 데이터가 커질수록 감당할 수 없게 됩니다.

## DeukPack의 스마트한 분리 시트 (SplitSheet)

DeukPack은 IDL에서 `list<T>`나 `map<K, V>`를 선언하면, 이를 강제로 콤마로 묶지 않고 **엑셀의 시트(Sheet)를 분리**하여 관계형 DB처럼 다루는 강력한 기능을 제공합니다.

```deuk
record Monster {
  1> int64 id
  2> list<DropItem> dropItems // 기본적으로 [Excel(SplitSheet)] 적용
}
```

DeukPack 엑셀 프로토콜은 메인 시트(`Monster`)와 분리 시트(`2 lst dropItems`)를 자동으로 엮어냅니다.
분리된 시트에는 부모의 `meta_id`가 자동으로 복사되어 있어, 기획자는 관계형 DB 테이블에 데이터를 넣듯 직관적으로 1:N 데이터를 여러 줄에 걸쳐 편집할 수 있습니다.

## tablelink를 통한 참조 무결성

나아가 DeukPack의 `tablelink<Table, Key>` 키워드는 "이 필드의 숫자는 저 테이블의 ID를 가리킨다"는 외래키(Foreign Key) 역할을 합니다.
엑셀 빌드 시, DeukPack 파이프라인은 `tablelink`가 걸린 필드의 값이 실제 대상 테이블에 존재하는지(참조 무결성) 자동으로 싹 훑어서 검증해 줍니다.

## 결론: 엑셀을 관계형 DB처럼 우아하게

배열 처리를 위해 엑셀을 누더기로 만들 필요가 없습니다. DeukPack은 복잡한 자료구조(List, Map)를 엑셀 시트로 깔끔하게 매핑하고, 연결 고리가 끊어지지 않도록 시스템 차원에서 완벽하게 방어해 줍니다.

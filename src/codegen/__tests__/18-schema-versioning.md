# 18편: 스키마 버전 관리와 호환성: 필드 추가/삭제에도 클라이언트가 죽지 않는 이유

라이브 서비스 중인 게임에 새로운 기능(예: 펫 시스템)이 추가되었습니다. 서버 개발자는 패킷 구조체에 `int32 petId` 필드를 추가하고 서버를 배포했습니다. 
그런데 아직 스토어에서 앱 업데이트를 받지 않은 구버전 클라이언트 유저들이 접속하면 어떻게 될까요?

JSON을 쓰거나 엉성한 바이너리 직렬화를 쓴다면, 파싱 오류가 발생하며 구버전 유저들의 화면이 멈추거나 튕길 것입니다.

## 와이어 포맷의 강력함: 알 수 없는 필드 건너뛰기

DeukPack(DpBinaryProtocol 기반)은 구조체의 필드 번호(ID)와 데이터 타입(Type)을 값과 함께 바이트 스트림에 새겨 넣습니다.
이 구조적 특징 덕분에 **완벽한 상위 호환성(Forward Compatibility)**과 **하위 호환성(Backward Compatibility)**을 보장합니다.

### 구버전 클라이언트가 신버전 패킷을 받을 때
서버가 보낸 바이트 스트림에 클라이언트가 모르는 필드 번호(예: `petId` 필드 번호 5번)가 들어있다면?
DeukPack의 파서는 크래시를 내지 않습니다. 헤더에 명시된 데이터 타입을 보고 해당 필드의 크기만큼 **안전하게 건너뛰고(Skip)**, 자신이 아는 필드들만 정확히 역직렬화합니다.

### 신버전 클라이언트가 구버전 패킷을 받을 때
새로 추가된 필드 데이터가 서버로부터 오지 않았다면? 
DeukPack은 해당 필드를 `.deuk` IDL 파일에 명시된 **기본값(Default Value)**으로 조용히 채워 넣습니다.

## 안전한 스키마 진화 규칙

DeukPack 환경에서 서버 무정지 업데이트를 달성하기 위한 규칙은 단 두 가지뿐입니다.
1. 이미 배포된 필드의 **번호(ID)를 바꾸거나 재사용하지 말 것.**
2. 필요 없어진 필드는 `[Obsolete]` 처리하거나 남겨두고, 새로운 필드는 항상 **새로운 번호**로 추가할 것.

## 결론: 강제 업데이트의 종말

DeukPack의 엄격하지만 유연한 와이어 포맷 구조 덕분에, 서버와 클라이언트의 버전이 일치하지 않아도 애플리케이션은 죽지 않습니다. 개발팀은 클라이언트 강제 업데이트 패치라는 극단적인 수단 없이도 훨씬 자유롭고 빈번하게 기능을 배포할 수 있습니다.

# 19편: SQLite와 DeukPack을 활용한 암호화된 게임 로컬 데이터 시스템 구축

게임의 기획 데이터(아이템 스탯, 몬스터 정보 등) 수십 MB를 모바일 클라이언트에 내려받을 때, 이를 거대한 JSON 파일 한두 개로 묶어서 보내는 방식은 치명적인 단점들을 가집니다.

1. **메모리 폭발:** 50MB짜리 JSON을 파싱하려면 메모리상에 수백 MB의 가비지가 순간적으로 치솟습니다(OOM 크래시의 주범).
2. **보안 취약점:** 유저가 파일 관리자 앱으로 JSON을 열어보고 마음대로 변조(핵)할 수 있습니다.
3. **느린 로딩:** 게임을 켤 때마다 전체 데이터를 메모리에 다 올려야 합니다.

## DeukPack의 SQLite 통합 파이프라인

DeukPack은 이 문제를 해결하기 위해, 메타데이터 팩을 생성할 때 JSON이나 단순 바이너리를 넘어 **SQLite DB 파일(.db)**로 다이렉트 출력을 지원하는 구조를 제안합니다.

### 1. 구조적 정합성
DeukPack의 IDL(`.deuk`)에 선언된 구조체들은 SQLite의 테이블(Table) 스키마와 완벽하게 1:1로 매핑됩니다. 앞서 다뤘던 분리된 List나 Set 데이터도 정규화된 자식 테이블로 깔끔하게 저장됩니다.

### 2. O(1) 수준의 필요 지연 로딩 (Lazy Loading)
유저가 10,000번 몬스터를 만났을 때, 전체 데이터를 메모리에 올릴 필요가 없습니다. Entity Framework나 로컬 쿼리를 통해 SQLite에서 `id = 10000`인 행 하나만 꺼내 DeukPack 객체로 즉시 변환합니다. 초기 로딩 시간과 메모리 점유율이 획기적으로 줄어듭니다.

### 3. 암호화 기반 보안 (SQLCipher)
가장 강력한 이점입니다. DeukPack의 출력 파이프라인을 SQLCipher와 결합하면, 기획 데이터가 담긴 SQLite 파일 전체가 AES-256으로 강력하게 암호화됩니다. 클라이언트 코드 내부에 숨겨진 비밀번호를 통해서만 DB를 열고 쿼리할 수 있으므로, 클라이언트 변조 해킹을 원천적으로 차단할 수 있습니다.

## 결론: 모바일 최적화의 종착지

수많은 데이터를 다뤄야 하는 현대 모바일 게임에서 메모리와 보안 관리는 핵심입니다. DeukPack의 스키마를 SQLite 파이프라인과 결합하는 것은, 가장 안전하고 메모리 효율적인 로컬 데이터베이스를 구축하는 궁극의 해법입니다.
